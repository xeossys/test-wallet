# XEOS Network

**Version:** 1.0.0
**Architecture:** Python / Flask / SQLite / Ed25519 Cryptography
**Status:** Production-Ready Lightweight Ledger

---

## Overview

XEOS Network is a localized, cryptographically secure decentralized ledger system. Built upon Python and Flask, it utilizes Ed25519 asymmetric cryptography to facilitate secure, immutable peer-to-peer transactions. 

The application operates an in-memory consensus engine that reconstructs state by evaluating an immutable SQLite-backed blockchain. It includes a modern, responsive frontend powered by Tailwind CSS, enabling users to authenticate, manage their digital portfolios, and securely execute and broadcast cryptographic payloads.

---

## System Architecture

The XEOS infrastructure is divided into four core operational layers:

1.  **Cryptographic Identity Management:** Automated generation of Ed25519 key pairs. Aliases (`@username`) are persistently mapped to 64-character public wallet addresses.
2.  **Consensus Engine:** A deterministic state machine that reads raw transaction payloads from the SQLite database, verifies cryptographic signatures in real-time, and calculates absolute network balances.
3.  **Transaction Manifests:** Transactions are generated as signed JSON manifests rather than direct database updates. These manifests can be transmitted out-of-band and claimed securely by the recipient, preventing replay attacks.
4.  ** Frontend:** A seamless, glassmorphism-styled GUI served directly through Flask, providing global synchronization polling, clipboard management, and manifest generation.

## Installation & Deployment

### Prerequisites

Ensure the following dependencies are installed on your host environment:
* Python 3.8 or higher
* `pip` package manager

### Environment Setup

1.  **Clone or Download the Source Code**
    Place the XEOS script in your designated application directory.

2.  **Install Required Libraries**
    Execute the following command to install the necessary Python packages:
    ```bash
    pip install flask cryptography
    ```

3.  **Configure the Central Mint (Admin Key)**
    Locate the configuration section at the top of the script. To enable the issuance of genesis bonds (minting initial currency), you must input your network's primary Ed25519 Public Key.
    ```python
    # =====================================================================
    # PASTE YOUR CENTRAL MINT RED PUBLIC KEY HERE
    ADMIN_PUBKEY = "your_64_character_hex_public_key_here"
    # =====================================================================
    ```

4.  **Launch the Network Node**
    Initialize the application. The system will automatically construct the local `xeos_.db` ledger if it does not exist.
    ```bash
    python xeos.py
    ```
    *Note: The system binds to `0.0.0.0` and defaults to port `5000`, or utilizes the port specified in your environment variables, making it compatible with cloud deployment platforms like Render, Railway, or Heroku.*

## Operational Guide

### Network Authentication
Navigate to the hosted IP address or domain. The network utilizes a seamless registration system. Entering an unregistered `@username` will automatically generate a new cryptographic key pair and bind it to the alias. Returning users are mapped to their existing keys.

### Executing Transfers
1. Access the **Send Funds** interface.
2. Specify a destination (either an `@username` or a 64-character raw address).
3. Input the exact transfer amount.
4. Click **Sign & Broadcast**.
5. The system will generate a cryptographic manifest (JSON block). Provide this raw manifest to the recipient.

### Claiming Cryptographic Bonds & Transfers
1. Access the **Claim Bond** interface.
2. Paste the exact JSON manifest generated by the sender (or the Central Mint).
3. Click **Verify Cryptography**.
4. The consensus engine will verify the Ed25519 signatures, ensure the recipient matches the authenticated session, check against replay attacks, and permanently write the transaction to the global ledger.

## Application Programming Interface (API)

The XEOS node provides the following internal REST endpoints to facilitate the frontend GUI:

| Endpoint | Method | Function |
| :--- | :--- | :--- |
| `/api/auth` | `POST` | Authenticates or registers a user alias. Generates backend key pairs. |
| `/api/logout` | `GET` | Clears the current session token. |
| `/api/state` | `GET` | Returns the absolute network balance for the active session address. |
| `/api/send` | `POST` | Validates balances, generates an Ed25519 signature, and returns the JSON manifest. |
| `/api/claim` | `POST` | Executes cryptographic verification and commits manifests to the immutable chain. |

## Security Specifications

* **Signature Algorithm:** Ed25519 (EdDSA)
* **Session Management:** Secure cryptographic browser sessions via `os.urandom(24)` keys.
* **Replay Protection:** Manifest signatures act as primary keys in the database schema; duplicates are mathematically rejected.
* **Data Integrity:** The consensus algorithm relies exclusively on raw, verified chain data to calculate balances, negating database manipulation attacks on user balances.

> **Important Security Advisory:** In this specific lightweight configuration, private keys are generated and stored within the central SQLite database (`xeos_.db`) to facilitate seamless web interactions. In a zero-trust, production-grade blockchain environment, private keys must remain strictly client-side. Protect the database file utilizing strict OS-level permissions.
